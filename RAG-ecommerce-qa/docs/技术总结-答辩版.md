### 架构分析

- API 可以被多个客户端复用（Web、App、小程序）
- 便于后期扩展为微服务架构

**为什么选择 FastAPI？**
- 原生支持异步（async/await），高并发性能好
- 自动生成 OpenAPI 文档（Swagger UI）
- 类型提示支持，代码更规范

### 🎓 模拟答辩问题

**Q1：为什么选择单机部署而不是分布式？**

> **回答：** 本项目是学习和演示用途，单机部署更简单直接。但架构设计上是模块化的，向量数据库和 LLM 都是独立组件，后期可以轻松替换为分布式方案（如 Milvus 替换 ChromaDB，DeepSeek 本地部署替换 API 调用）。

### RAG 流程分析

**完整流程（api.py 第 240-285 行）：**

```python
# ① 问题向量化

# ② 向量检索

    query_embeddings=[query_embedding.tolist()],

        sources.append({...})

# ⑤ 返回结果
```

### 🎓 模拟答辩问题

**Q1：什么是 RAG？为什么需要 RAG？**

> **回答：** RAG 是检索增强生成（Retrieval-Augmented Generation）。传统 LLM 有两个问题：一是知识截止日期，二是会产生幻觉（编造答案）。RAG 通过先检索知识库获取真实信息，再让 LLM 基于这些信息生成回答，既保证了准确性，又保持了自然语言的流畅性。

**Q2：RAG 和直接用 LLM 有什么区别？**

> **回答：** 直接用 LLM，它会凭空回答，可能编造错误信息。RAG 先从知识库检索真实文档，再让 LLM 基于这些文档回答。比如用户问"什么时候发货"，直接用 LLM 可能说"3-5天"，但 RAG 会检索到我们的真实政策"48小时内发货"，然后生成准确回答。

**Q3：为什么不直接返回检索结果，还要调用 LLM？**

> **回答：** 检索结果是原始的 Q&A 对，比较生硬。LLM 可以把这些信息组织成自然、友好的客服语言。比如检索到"48小时内发货"，LLM 会生成"亲，我们会在48小时内为您安排发货哦~"，用户体验更好。

### 向量化流程分析

**embed_query() 函数（embedder.py 第 190-212 行）：**

```python
def embed_query(self, query: str) -> np.ndarray:
    # BGE 模型查询端需要加前缀
    if not query.startswith("为这个句子生成表示以用于检索相关文章："):
        query = "为这个句子生成表示以用于检索相关文章：" + query

    embedding = self.model.encode(
        [query],
        normalize_embeddings=True
    )
    return embedding[0]
```

**为什么加前缀？**
- BGE 模型训练时，查询端和文档端使用不同的编码方式
- 查询端加前缀可以让查询向量和文档向量在同一语义空间
- 这是 BGE 论文的设计约定，能提升检索精度

### 🎓 模拟答辩问题

**Q1：什么是向量化？为什么要向量化？**

> **回答：** 向量化就是把文字变成一组数字（向量），让计算机能计算语义相似度。比如"发货"和"快递"意思相近，向量化后它们的向量距离很远；而"发货"和"天气"意思不同，向量距离很远。这样我们就能用数学方法找到语义最相似的文档。

**Q2：为什么选择 BGE 模型？**

> **回答：** BGE 是智源研究院发布的中文向量模型，在中文语义检索任务上效果很好。bge-small 是小模型（24M参数），速度快，适合实时问答场景。如果需要更高精度，可以换成 bge-large。

**Q3：什么是降级方案？为什么需要？**

> **回答：** 降级方案是在模型加载失败时的备选方案。比如网络不好下载不了模型，或者 GPU 内存不够，系统会用 MD5 哈希生成伪向量。虽然精度很低，但至少系统不会崩溃，用户能看到一个基本回答。这叫"优雅降级"。

```python
def search(self, query_embedding, top_k=5):
    # 查询向量库
    results = self.collection.query(
        query_embeddings=[query_list],
        n_results=top_k
    )

    # 格式化结果
    for i in range(len(results['documents'][0])):
        result = {
            'text': results['documents'][0][i],
            'score': 1 - results['distances'][0][i],  # 距离转相似度
            'metadata': results['metadatas'][0][i]
        }
        formatted_results.append(result)

    return formatted_results
```

### 🎓 模拟答辩问题

**Q1：什么是向量数据库？和传统数据库有什么区别？**

> **回答：** 传统数据库存的是结构化数据（文字、数字），用精确匹配查询。向量数据库存的是向量（高维数字），用相似度搜索。比如查"发货"，传统数据库只能找到完全包含"发货"的记录；向量数据库能找到"快递"、"物流"等语义相近的记录。

**Q2：为什么选择 ChromaDB？**

> **回答：** ChromaDB 是轻量级向量数据库，Python 原生支持，不需要额外部署服务，数据存在本地文件里。适合学习和中小规模项目。如果是生产环境大规模数据，可以换成 Milvus 或 Qdrant。

**Q3：什么是 HNSW 索引？**

> **回答：** HNSW 是层级可导航小世界图，是一种高效的近似最近邻搜索算法。它把向量组织成多层图结构，搜索时从顶层开始逐层向下，快速定位到最相似的向量。时间复杂度是 O(log n)，比暴力搜索 O(n) 快很多。

### 🎓 模拟答辩问题

**Q1：为什么要分块？直接存整篇文档不行吗？**

> **回答：** 分块有两个原因：一是向量模型有长度限制（通常 512 token），太长的文本无法向量化；二是分块后检索更精确，能找到最相关的段落，而不是返回整篇文档。

**Q2：为什么要重叠？重叠多少合适？**

> **回答：** 重叠是为了避免在句子中间切断，丢失上下文。比如一个句子被切成两半，向量化后语义就不完整了。重叠 50 字可以保证每个块都有完整的上下文。重叠太多会浪费存储，太少会丢失信息，50 字是比较平衡的值。

### 🎓 模拟答辩问题

**Q1：为什么选择 DeepSeek？**

> **回答：** DeepSeek 是国产大模型，中文理解能力强，API 价格便宜，响应速度快。而且它兼容 Anthropic 接口格式，接入方便。相比 OpenAI，DeepSeek 在中文场景下效果更好，成本更低。

**Q2：什么是提示词工程？**

> **回答：** 提示词工程就是设计给 LLM 的输入，让它按照我们的要求回答。比如我们告诉 LLM"你是电商客服"、"基于参考信息回答"、"不要编造"，这样 LLM 就会生成符合客服场景的回答，而不是通用的百科式回答。

**Q3：LLM 失败时怎么办？**

> **回答：** 我们有容错机制。如果 LLM 调用失败（网络问题、API 限流等），系统会自动回退到知识库的模板回答。虽然不如 LLM 生成的自然，但至少用户能得到答案，不会看到错误页面。

### 🎓 模拟答辩问题

**Q1：如何优化检索速度？**

> **回答：** 三个优化：一是缓存机制，相同文本不重复向量化；二是 HNSW 索引，O(log n) 复杂度；三是懒加载，模型只在第一次请求时加载，启动速度快。

**Q2：如何处理高并发？**

> **回答：** FastAPI 基于 ASGI，支持异步处理。向量检索和 LLM 调用都是非阻塞操作，一个请求等待 LLM 响应时，其他请求可以继续处理。如果需要更高并发，可以用 Gunicorn 多 worker 部署。
